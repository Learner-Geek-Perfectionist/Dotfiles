# Skill 构想：深度代码审查（Deep Code Review）

> 创建日期：2026-03-29
> 状态：构想阶段，待实现
> 使用 `example-skills:skill-creator` 创建

## 问题背景

全量代码审查（如 2026-03-28 的 156 issue 报告）能发现大量表面缺陷，但会遗漏"过度工程"等需要设计判断的深层问题。原因：

1. **广度扫描牺牲深度** — 60 文件一次性扫，没有余力做上下文跳转
2. **审查维度缺失** — "代码精简度"被理解为"找重复"，没有"质疑存在"视角
3. **LLM 默认尊重已有代码** — 不会主动问"这段代码该不该存在"

## 设计思路

### 输入

指定要审查的文件或模块（避免全量扫描），例如：
- 单文件：`lib/utils.sh`
- 模块：`scripts/install_*.sh`
- 最近修改：`git diff --name-only HEAD~5`

### 三轮分层审查

| 轮次 | 视角 | 核心问题 | 方法 |
|---|---|---|---|
| 第一轮：设计审查 | 质疑存在 | 该不该存在？复杂度合理吗？ | 对每个 >15 行函数做需求回溯：谁调用？调用频率？精度/复杂度与使用场景匹配吗？ |
| 第二轮：正确性审查 | 找 bug | 实现对不对？ | 边界情况、未初始化变量、死代码、安全漏洞 |
| 第三轮：精简审查 | 消除冗余 | 有重复模式吗？能提取公共逻辑吗？ | 跨文件比对相似代码块，建议提取通用函数 |

### 第一轮的关键 prompt 要素

- "对每个超过 15 行的函数，回答：谁调用了它？调用多少次？当前实现的复杂度和实际使用场景匹配吗？"
- "如果一个函数只被调用 1-2 次，且调用场景不需要其全部能力，标记为'疑似过度工程'"
- "对辅助函数，追问：它服务的主函数值得这个辅助函数存在吗？"

### 输出格式

按严重程度分类，每个 issue 包含：
- 问题描述
- 调用链分析（谁调用了它、调用频率）
- 简化建议（如有）
- 当前行数 vs 建议行数

## 实际案例

`print_banner`（32 行 + `_display_width` 辅助函数）→ 只被调用 1 次，简化到 13 行纯 bash，精度不变。这类问题在全量审查中被遗漏，但在设计审查轮中会被直接发现。
